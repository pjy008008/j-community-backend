# J-Community - Backend (Spring Boot API Server)

> **"프론트엔드 개발자가 협업하기 쉬운 백엔드 서버"**

풀스택 개발자로의 성장을 목표로, **Python(Django/FastAPI)** 경험을 넘어  
**Spring Boot**를 통해 견고하고 신뢰할 수 있는 API를 구축하는 백엔드 프로젝트입니다.

## 1. 새로운 도전

이 프로젝트는 **새로운 도전**과 **몰입**이라는 두 가지 키워드에서 시작되었습니다.

기존에 익숙했던 프론트엔드 개발이나 Python 기반의 백엔드(`Django`, `FastAPI`)를 넘어,  
**Java와 Spring 생태계에 깊이 몰입**하여 실제 프로젝트를 완성도 있게 구현해보고자 했습니다.

**Spring을 선택한 이유**
- **객체 지향 설계:** 프리코스에서 익힌 Java 문법과 객체 지향 설계를 실제 프로젝트에 적용해보기 위해
- **‘풀스택’의 완성:**  
  프론트엔드 개발자의 관점에서 “어떻게 백엔드를 구현해야 프론트엔드와 협업하기 편할까?”라는 고민을 직접 해결하고,
  양쪽 모두에 도움 되는 코드를 작성하는 개발자가 되기 위함

## 2. 프로젝트의 핵심 목표
 
> 단순히 돌아가는 백엔드가 아니라,  
> 프론트엔드 개발자가 **“협업하기 쉬운”** 백엔드를 만드는 것.

단순히 기능이 동작하는 것을 넘어, **협업의 품질을 높이는 백엔드**를 지향합니다.

### 깨진 유리창 이론과 백엔드의 역할

이 프로젝트는 **Reddit**과 같이 “잘 정돈된 커뮤니티”를 지향하며,  
UI의 시각적 신뢰도가 사용자의 행동에 긍정적인 영향을 미친다는  
**깨진 유리창 이론(Broken Windows Theory)** 을 핵심 철학으로 삼습니다.

백엔드는 이 철학의 **보이지 않는 기반**입니다.

> UI가 아무리 깔끔해도,  
> API 응답이 느리거나, 데이터 구조가 일관성 없거나,  
> 에러 처리가 불명확하면 사용자는 “깨진 유리창”을 경험하게 됩니다.

백엔드가 **견고하고 예측 가능하며 신뢰할 수 있는 데이터를 제공**할 때,  
프론트엔드는 비로소 **잘 정돈된 UI**와 **신뢰도 높은 사용자 경험**을 구축할 수 있습니다.

## 3. 기술적 목표

**협업하기 쉬운 백엔드**를 만들기 위해 다음 원칙을 준수합니다.

- 명확한 API 명세 
  - API의 요청(Request)과 응답(Response) 형식을 명확하게 정의합니다.
  - 프론트 개발자가 문서를 보며 “이 API는 어떤 데이터를 주고, 어떤 값을 보내야 하는지”를 즉시 파악할 수 있도록 합니다.

- 일관성 있는 데이터 구조
  - 데이터베이스의 Entity를 API 응답으로 직접 노출하지 않습니다.
  - 프론트엔드에서 사용하기 편한, **일관된 DTO(Data Transfer Object)** 로 가공하여 전달합니다.

- 친절한 에러 핸들링
  - `500 Internal Server Error`와 같이 모호한 응답을 지양합니다.
  - “이메일 중복”, “권한 없음” 등 프론트엔드가 즉시 사용자에게 피드백할 수 있는 **명확한 에러 코드와 메시지**를 반환합니다.
